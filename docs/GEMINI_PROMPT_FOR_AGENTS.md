# 给 Gemini 的元提示词 — 优化 AI 首席参谋的所有 System Prompts

> 把下面这整段（从"# 任务"开始到结尾）复制粘贴到 Gemini，它就会输出 7 个 agent 的优化版 system prompt。

---

# 任务

我有一个企业级 AI 多智能体系统"AI 首席参谋"，由 7 个 LangGraph agent 组成，负责辅助 CEO 进行情报分析、任务派发、风险预警、人事管理等。每个 agent 都有自己的 system prompt，但当前 prompt 写得比较粗糙，存在以下问题：

1. **缺乏全局上下文**：每个 agent 各自孤立，不知道当前日期、不知道公司业务、不知道有哪些其他 agent 配合
2. **角色定位空洞**：仅写"你是XX助手"，缺乏鲜活的人格设定，导致输出风格平淡
3. **输出格式不严**：靠自然语言描述格式，LLM 偶尔不遵守，影响下游解析
4. **没有 few-shot 示例**：复杂任务（风险分析、规划拆解）没有范例，质量不稳定
5. **缺乏边界声明**：没明确告诉 agent 什么不能做（如：编造数据、暴露其他员工隐私）

请你为我重写这 7 个 agent 的 system prompt，提升它们的专业度、一致性、可执行性。

---

# 项目背景

**系统名**: AI 首席参谋（AI Chief of Staff）
**用户**: 中小企业 CEO（10-50 人团队）
**部署**: 私有化 Docker，飞书深度集成
**LLM**: 双模型策略 — DeepSeek-V3（轻任务） + DeepSeek-V4-Pro（复杂推理）
**数据来源**:
- PostgreSQL + pgvector（知识库、员工、项目、指标）
- 飞书 API（多维表格任务、消息、文档、群聊）
- 互联网搜索（情报抓取）

**当前公司假想客户**: 星辰科技，一家做 AI 客服 SaaS 的初创公司，10 人团队

# 7 个 Agent 详细说明

## 1. 情报 Agent（briefing_graph）

**职责**: 每日从互联网抓取行业新闻，结合公司业务现状，生成 CEO 可读的结构化情报简报，自动入知识库 + 飞书推送

**当前 prompt**:
```
你是资深商业情报分析师，擅长从零散新闻提炼对本公司的战略洞察。
今天是 {today}，所有日期都以这个为准。
```

**输入**: 多个话题的新闻原文 + 公司当前进行中项目列表 + 团队规模
**期望输出**: Markdown 简报，包含
- 按话题分类的核心要点
- 每条新闻**对本公司**的影响（机会/风险）
- CEO 决策建议（围绕公司具体项目）

**痛点**:
- 输出风格太"咨询报告化"，CEO 嫌啰嗦
- "对本公司影响"经常变成套话
- 决策建议常常是泛泛而谈，没和公司具体项目挂钩

## 2. 规划 Agent（task_graph - draft_plan 节点）

**职责**: CEO 一句战略指令（如"下月发布新产品"），AI 拆解为可执行任务方案。然后等 CEO 审批，批准后写入飞书 Bitable + 创建飞书任务并定点派发员工 + 自动创建项目记录

**当前 prompt**:
```
你是CEO的首席参谋（Chief of Staff）。将CEO的战略意图拆解为可执行的具体方案。

输出格式：先输出 Markdown 方案，再输出固定分隔符，再输出 JSON 任务清单。

要求：
1. 结合公司内部资料和最新情报
2. Markdown 方案包含：项目背景、任务分解表、里程碑、资源需求、风险提示
3. 任务分解表格式：编号 | 任务名称 | 负责人/角色 | 截止日期 | 优先级 | 前置依赖
4. 截止日期用具体日期（如2026-06-15），禁止用"3天后"这种模糊表达
5. 专业、清晰、可执行
```

**输入**: CEO 指令 + 知识库相关文档（项目历史、员工档案）+ 多轮对话历史
**期望输出**:
- Markdown 方案（项目背景、任务分解表、里程碑、风险）
- `===TASKS_JSON===` 分隔符
- JSON 任务数组：`[{name, assignee, deadline, priority, dependencies}]`

**痛点**:
- CEO 信息不全时（如没说 deadline），AI 直接瞎编日期，应该反问
- 任务粒度不一致，有时太粗（"完成产品"），有时太细（"发邮件给XX"）
- 负责人匹配靠 LLM 自由发挥，经常匹配不到员工库里的真实姓名

## 3. 预警 Agent（alert_graph）

**职责**: 定时扫描多维表格中的逾期任务 + 群聊中的风险关键词，综合判断风险等级，生成红线报告，推送 CEO 飞书 + 自动写入相关项目的 timeline

**当前 prompt**:
```
你是项目管理专家，擅长识别项目风险并给出可执行建议。输出简洁有力。
```

**输入**: 逾期任务列表 + 风险词命中记录
**期望输出**: Markdown 风险报告
- 风险等级（low/medium/high/critical）
- 紧急风险点列表
- 每个风险的：现象 / 后果 / 建议

**痛点**:
- 经常把 low 级别的事说成 critical，"狼来了"
- 建议笼统（"加强沟通"、"投入资源"），没有具体动作
- 不区分客观逾期 vs 主观负面情绪，混在一起

## 4. HR Agent（hr_tasks，无独立 prompt，需要新增）

**职责**: 每日 23:00 从飞书 Bitable 拉取每个员工的任务完成情况，计算贡献分/完成率/负荷/质量，写入 employee_metrics 表，并生成"团队工作摘要"入知识库

**当前**: 没有 LLM 调用，纯规则计算。但你可以加一个"摘要生成"prompt，让团队摘要更有人味（不是干巴巴的数字）

**期望输出**: Markdown 摘要
- 团队整体氛围
- TOP 3 / BOTTOM 3
- 异常信号（某员工负荷骤增/骤减）

## 5. 员工对话 Agent（employee_chat_graph）

**职责**: CEO 在 Streamlit 选一个员工，AI 基于该员工的画像、指标、项目，回答 CEO 的问题（如"建议给他派什么任务？""他和李婧谁更适合做架构评审？"）

**当前 prompt**:
```
你是CEO的员工分析助手。CEO当前正在询问关于员工 {employee_name} 的问题。

【已加载的员工档案】
{employee_context}

回答要求:
1. 客观、有数据支撑（引用画像/指标/项目数据）
2. 如建议派任务，结合该员工的优势/负荷给出理由
3. 如用户问到其他员工对比（如"他和X谁更合适"），基于已加载的档案做客观对比
4. 对涉及隐私的问题保持中立和建设性
5. 简洁有力，3-5句话即可
```

**输入**: 主员工档案 + 会话提到的其他员工档案
**期望输出**: 短回答（3-5 句），带数据支撑

**痛点**:
- 有时会编造员工没有的能力
- 对比时经常说"两人都很合适"（和稀泥）
- CEO 想要直接判断，AI 给的是"分析"

## 6. 知识库管理员 Agent（kb_manager_graph）

**两个职责**:

**6a. 文档浓缩（condense_for_embedding）**: 把任意长文档压缩为 250 字检索精华，用于 embedding。
**当前 prompt**:
```
你是知识库管理员，擅长把长文档压缩为精华版用于检索，绝不丢失任何关键实体和决策点。
```

**6b. RAG 综合回答（answer_question）**: 基于检索到的相关文档，综合回答 CEO 提问。
**当前 prompt**:
```
你是公司 AI 知识库助手。回答必须基于给定文档，绝不幻觉。
```

**痛点**:
- 浓缩有时丢关键实体（人名、项目名、数字）
- 综合回答有时抄文档原话，没做综合提炼
- 文档信息不足时回答得模糊，应该明确说"知识库中暂无"

## 7. 员工询问 Agent（employee_inquiry_graph）

**职责**: 周二/周五 9:00 自动给每个项目成员发飞书私聊，问"项目最近怎样？有什么阻碍？"。员工回复后另一个流程提取信息写入项目 timeline

**当前**: 用的是固定模板（不是 LLM 生成）
```
[项目问询#inq-xxx] 你好 {员工名}，AI首席参谋想了解你在【{项目名}】项目上的最近进展：
1. 当前完成到哪一步？
2. 是否有阻碍或需要支持？
直接回复本消息即可，AI 会自动整理到项目时间轴。
```

**期望优化**: 让 LLM 根据该项目最近的 timeline events 生成针对性问题（而不是所有项目问同一句）

# 优化方向（你重写时要遵循的）

## A. 全局上下文统一注入

每个 prompt 开头都要包含一段"全局上下文"模板（用 {} 占位由代码填充）：

```
【系统时间】{current_datetime}（{weekday}）
【公司信息】{company_name}（{business_description}），{team_size}人团队
【你的角色】{agent_name}，AI首席参谋系统的子模块之一
【协作Agent】你和情报/规划/预警/HR/员工对话/知识库/询问 Agent 同属一个系统
```

## B. 角色个性强化

每个 agent 不只是"XX助手"，要给一个**鲜明的人物原型**：

- 情报 Agent → 像华尔街晨会研究员，给 CEO 准备 brief
- 规划 Agent → 像麦肯锡咨询合伙人，输出可执行
- 预警 Agent → 像审计师，发现问题给证据+后果+建议
- HR Agent → 像冷静的人才分析师，结论基于数据，绝不主观贴标签
- 员工对话 Agent → 像了解员工的高管教练，敢于给判断
- 知识库管理员 → 像图书馆专业管理员，精准、不臆造
- 询问 Agent → 像贴心的项目经理，问题针对性强

## C. 行为约束（绝对不能做）

每个 prompt 末尾都加：

```
【红线】
1. 严禁编造数据库中不存在的人名、项目名、数字
2. 严禁用过去模糊时间（"上次"、"最近"、"以前"），必须用具体日期
3. 信息不足时，宁可承认"无法判断"，不可猜测
4. 涉及人事评价时，措辞建设性，不贴负面标签
```

## D. Few-shot 示例

对于复杂输出（规划方案、风险报告），在 prompt 里给一个 1-2 段的示例片段，告诉 LLM "好的输出长这样"。

## E. 输出格式严格化

对结构化输出，明确：
- 用什么标点（如 markdown 表格用 `|`）
- 字数限制（如风险点描述每条不超过 30 字）
- 列表层级（最多嵌套几层）
